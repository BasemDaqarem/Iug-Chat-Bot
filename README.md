# IUG Chatbot — شات بوت الجامعة الإسلامية بغزة

شات بوت عربي يعتمد على تقنية RAG (استرجاع + توليد): يقرأ بيانات الجامعة من MongoDB،
يبني فهرسًا دلاليًا بواسطة Jina Embeddings، ويجيب على أسئلة الطلاب عبر Groq LLM
مع حماية خصوصية بيانات الطلاب (المعدلات والترتيب).

## البنية المعمارية

```
Chatbot_IUG/
├── app/                    ← الحزمة الأساسية (كل وحدة = مسؤولية واحدة)
│   ├── config.py           ← كل الإعدادات وقراءة .env (المكان الوحيد)
│   ├── prompts.py          ← قوالب الـ system prompts العربية
│   ├── db.py               ← اتصال MongoDB موحّد (القاعدة الرئيسية + قاعدة الملفات)
│   ├── chunking.py         ← تحويل الوثائق إلى مقاطع نصية + كشف الوثائق الحساسة
│   ├── text_norm.py        ← تطبيع عربي + تقطيع كلمات (للبحث اللفظي)
│   ├── lexical.py          ← مُحرّك BM25 اللفظي (بلا اعتماديات خارجية)
│   ├── embeddings.py       ← عميل Jina + بناء الفهرس الدلالي
│   ├── index_store.py      ← حفظ/كاش الفهرس على القرص (إقلاع فوري، بلا إعادة حساب)
│   ├── retrieval.py        ← ترتيب دلالي + بحث هجين (دلالي + BM25 عبر RRF)
│   ├── llm.py              ← عميل Groq (بناء الطلب + إعادة المحاولة + الأخطاء)
│   ├── sessions.py         ← سجل المحادثات لكل جلسة (في الذاكرة)
│   ├── privacy.py          ← حارس الخصوصية + البحث عن سجل الطالب الحساس
│   ├── knowledge_base.py   ← دورة حياة قاعدة المعرفة + بحث مُرشَّح بالصلاحية
│   ├── uploaded_files.py   ← دورة حياة الملفات المرفوعة (فهرس مستقل لكل ملف)
│   ├── chatbot.py          ← IUGChatbot: واجهة موحّدة تنسّق بين الوحدات
│   └── api/                ← طبقة FastAPI (رقيقة — تفوّض كل شيء لـ IUGChatbot)
│       ├── __init__.py     ← create_app() — مصنع التطبيق + lifespan
│       ├── schemas.py      ← عقود Pydantic لكل طلب/رد
│       ├── deps.py         ← حقن التبعيات (البوت singleton عبر app.state)
│       ├── middleware.py   ← CORS + توقيت الطلبات + GZip
│       └── routers/        ← health / chat / files / sessions
├── server.py               ← نقطة تشغيل الـ API (uvicorn)
├── cli.py                  ← واجهة الاختبار من الطرفية (console)
├── chatbot_core.py         ← شِيم توافق خلفي (الاستيرادات القديمة تعمل كما هي)
├── database.py             ← شِيم توافق خلفي → app/db.py
├── uploaded_files_db.py    ← شِيم توافق خلفي → app/db.py
├── tests/                  ← اختبارات وحدة + اختبار تكافؤ مع النسخة الأصلية
└── requirements.txt
```

## تدفق البيانات (سؤال عادي)

```
سؤال الطالب
   │
   ▼
embeddings.embed_query ──► retrieval.rank_chunks (على فهرس knowledge_base)
   │
   ▼
chatbot.chat():
   1. بحث مُرشَّح بالصلاحية: المرشّحون = المقاطع العامة + سجل الطالب نفسه فقط
      (متجهات الطلاب الآخرين لا تدخل الحساب أصلًا — عزل بنيوي)
   2. بحث هجين (دلالي + BM25) على المرشّحين ثم دمج RRF
   3. اختصار "الحالة الأكاديمية" (إجابة مباشرة من سجل الطالب دون LLM)
   4. حارس الخصوصية (حجب أسئلة معدل/ترتيب طالب آخر دون LLM)
   5. بناء السياق: سجل الطالب نفسه + المقاطع العامة
   │
   ▼
llm.chat_completion (Groq) ──► sessions.push ──► الإجابة
```

## التشغيل

```bash
pip install -r requirements.txt
# عبّئ .env (انظر المفاتيح أدناه) ثم:

python server.py         # خدمة الويب (FastAPI) — التوثيق التفاعلي: http://127.0.0.1:8000/docs
python cli.py            # أو واجهة الطرفية للاختبار السريع
```

### نقاط الـ API

| النقطة | الوصف |
|---|---|
| `GET /health` | جاهزية الخدمة + إحصاءات قاعدة المعرفة والنماذج |
| `POST /api/chat` | محادثة على قاعدة المعرفة (body: `question`, `session_id`) |
| `POST /api/chat/files` | محادثة عبر كل الملفات المرفوعة (ترتيب مدموج) |
| `POST /api/chat/files/{name}` | محادثة على ملف واحد فقط |
| `GET /api/files` | قائمة الملفات المرفوعة وحالة فهارسها |
| `PUT /api/files/{name}` | رفع/استبدال ملف JSON (body: `documents`) |
| `POST /api/files/{name}/reload` | إعادة فهرسة ملف |
| `DELETE /api/files/{name}` | حذف ملف |
| `GET /api/sessions/{sid}/history` | سجل محادثة الجلسة |
| `DELETE /api/sessions/{sid}/history` | مسح سجل الجلسة |
| `GET /api/cache/stats` | إحصاءات الكاش (hit rate، الحجم، الإزالات) |
| `POST /api/cache/clear` | إبطال الكاش يدوياً |

إعدادات الـ API في `.env`: `API_ENV` (development/production — في production تُخفى `/docs`)،
`API_HOST`, `API_PORT`, `API_CORS_ORIGINS` (origins الواجهة مفصولة بفواصل).

## الكاش (Caching) — الخصوصية قبل الأداء

كاش داخلي (in-process، TTL + LRU، `app/cache.py`) بطبقتين آمنتين:

1. **كاش متجهات الأسئلة** (`embeddings.embed_query`): نص السؤال → متجهه من Jina.
   حتمي، والقيمة متجه (ليست إجابة ولا سجل)، فآمن ومشترك. يوفّر نداء Jina للأسئلة المكررة.
2. **كاش الإجابات العامة** (`chatbot`): يُخزَّن الرد الكامل **فقط** إذا كان الدور **عامًا** —
   أي الجلسة **لا تملك سجلًا حسّاسًا** و**بلا سجل محادثة سابق**. عندها تعتمد الإجابة على
   السؤال + المحتوى العام فقط، فتُشارَك بأمان.

**الضمان الأمني (Phase 6/7/8):** أي رد يعتمد على بيانات طالب خاصة (سجل، معدل، ترتيب)
**لا يُخزَّن ولا يُقرأ من الكاش أبدًا** — يُولَّد لحظيًا دائمًا. الطالب صاحب السجل لا يتلقى
إجابة مخزّنة من زائر. لا تُخزَّن أي بيانات خاصة بمستخدم، فلا تسرّب بين المستخدمين ممكن.
الكاش يُبطَل تلقائيًا عند رفع/حذف/إعادة فهرسة أي ملف. مراقبة عبر `GET /api/cache/stats`.

مقياس حي: سؤال عام مكرّر انتقل من **5.75s → 0.03s** (≈198×) بنفس الإجابة.

## قواعد التصميم

- كل وحدة في `app/` تملك مسؤولية واحدة فقط (SRP).
- `config.py` هو المكان الوحيد الذي يقرأ `.env` — لا `os.getenv` في أي مكان آخر.
- نصوص الـ prompts سلوكٌ وليست تفاصيل: أي تعديل عليها في `prompts.py` يغيّر الإجابات.
- منطق التقطيع في `chunking.py` يجب أن يبقى مطابقًا بايت-ببايت لما بُنيت عليه
  الـ embeddings الحالية (أي تغيير يستلزم إعادة فهرسة كاملة).
- الوحدات الهيكلية (`database.py`, `uploaded_files_db.py`, `chatbot_core.py`)
  شِيمات توافق فقط — الكود الجديد يستورد من `app/` مباشرة.

## ما أُنجز مؤخرًا (ترقيات RAG)

- ✅ **حفظ الفهرس على القرص** (`index_store.py`): إقلاع بلا إعادة حساب الـ embeddings،
  ويُعاد بناء الفهرس تلقائيًا فقط عند تغيّر المقاطع (بصمة model + النصوص).
- ✅ **بحث هجين** (`lexical.py` + `retrieval.hybrid_rank`): دمج الدلالي مع BM25 اللفظي
  عبر RRF — يرفع دقّة الأرقام/الرموز/الأسماء العربية، مع تطبيع عربي (`text_norm.py`).
- ✅ **عزل بيانات بنيوي**: `knowledge_base.search_for` يُرشّح بالصلاحية *قبل* الترتيب،
  فلا تدخل متجهات طالب آخر الحساب إطلاقًا (اختبارات `test_security.py`).
- ✅ **تقليل الهلوسة**: بوابة ثقة على النتائج (عتبة دلالية أو مطابقة لفظية فعلية).

## خارطة الطريق المتبقية

1. **المصادقة**: تسجيل دخول يتحقق من `students_auth` — حاليًا `session_id` غير موثّق،
   فأي قيمة تمرّ. **حرجة قبل أي إطلاق فعلي.**
2. **جلسات دائمة**: نقل `SessionStore` إلى Mongo/Redis (الآن تُفقد عند إعادة التشغيل).
3. **حارس خصوصية أقوى**: العزل البنيوي مُطبّق؛ يبقى تقوية كشف "سؤال عن طالب آخر"
   (المطابقة الآن بأول كلمة من الاسم) قبل أي إطلاق فعلي واسع.
4. **Vector DB (عند التوسّع فقط)**: تبنّي Qdrant/pgvector عند طبقة الويب متعددة العمّال
   وآلاف الطلاب المتزامنين — للتخزين المشترك والترشيح المحرّكي، لا للسرعة.
5. **واجهة أمامية (Frontend)** + نشر.

> كل وحدات `app/` تسجّل عبر `logging`؛ `cli.py` وحده يبقى بـ `print` لأنه واجهة الطرفية.
> يوجد `.env.example` نموذجًا لكل مفاتيح البيئة.
