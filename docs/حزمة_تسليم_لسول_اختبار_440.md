# حزمة تسليم لسول (Codex) — إعادة اختبار الـ440 على النسخة المحسّنة

**الغرض:** تعيد تشغيل بنك الـ440 سؤالاً (نفس ملفك الأصلي) على الكود المحسّن،
وتقارن قبل/بعد بالنتيجة **وبزمن الاستجابة**، لتحديد ما تحسّن وما ساء.
**تاريخ التسليم:** 18/7/2026.

## 0) تحذير حاسم قبل أي اختبار

**النسخة المرفوعة على GitHub (`origin/main` عند `532eec4`) قديمة ولا تحوي أي
تحسين.** التحسينات في **6 كوميتات محلية لم تُدفع بعد** لأن صلاحية الكتابة على
الريبو كانت مقفولة. الترتيب من الأقدم:

```
c0a72ef  خطة التحسين (تحقق مستقل من تقاريرك)
6171734  م1: وعي المرحلة + الاستبعاد + القوائم الكاملة + غير المحسوم
d8be9e8  م2+3: فاحص إجابة حتمي + Jina Reranker خلف فلاغ
a4163a6  سجل تنفيذ الخطة
9f5d891  مزوّد LLM احتياطي + قاعدتا حداثة المعلومة والفرع (من تشخيص الـ13)
a4c8691  إصلاح Q097 (تسريب الدراسات العليا) و Q093 (تحفّظ الطب)
```

**قبل الاختبار تأكد أن هذه الكوميتات وصلتك** (بعد أن يُعيد المالك فتح صلاحية
الكتابة ويدفعها محمود، أو بسحب الفرع المحلي). لو اختبرت `532eec4` ستقيس
النسخة القديمة وتظهر النتائج بلا تحسّن — وهذا خطأ منهجي لا فشل حل.

**البيانات الحيّة محدّثة فعلاً** على قاعدة Mongo (37 ملفاً، آخرها «معدل قبول
الطب» و«الدرجات والدراسات العليا والمنح» و«مسائل تحتاج تأكيد الجامعة») —
هذه لا تحتاج دفع كود، هي منشورة عبر لوحة الأدمن.

## 1) ثلاثة تصحيحات منهجية على تشغيلك الأول (إلزامية للعدالة)

هذه ليست انتقاداً — هي ما جعل تقريرك الأول يظلم أجزاءً من الحل:

1. **أطفئ الكاش:** `CACHE_ENABLED=false`.
   في تشغيلك الأول أُجيب Q036 في **0.168 ثانية** — زمن مستحيل لنداء LLM.
   كان **إصابة كاش** لجواب قديم سيّئ علق ساعة، فقِيسَ الكود القديم لا الجديد.
   أي بنشمارك بلا إطفاء الكاش = قياس ملوّث.

2. **احفظ `top_chunks` من الاستجابة.** قلتَ إن التمييز بين «فشل استرجاع»
   و«فشل توليد» يحتاج سجلاً غير متوفر — وهو **متوفر أصلاً**: حقل `top_chunks`
   يعود في كل استجابة API منذ البداية. احفظه لكل سؤال، فيصير التصنيف قطعياً.

3. **قِس الزمن لكل سؤال** (سجّل `latency_ms`) — هذا مطلب المقارنة الأساسي.

## 2) بيئة التشغيل الموصى بها

```env
CACHE_ENABLED=false          # إلزامي للبنشمارك
RERANK_ENABLED=false         # خط الأساس؛ فعّله في تشغيل ثانٍ منفصل للمقارنة
MAX_CONTEXT_CHARS=0          # أو 6000 على مفتاح صغير لتفادي 402
CHAT_API_MODEL=openai/gpt-oss-20b
# احتياطي تلقائي عند 402/429 (مضبوط في .env عندنا):
CHAT_FALLBACK_URL=https://api.sambanova.ai/v1/chat/completions
CHAT_FALLBACK_MODEL=gpt-oss-120b
```

- **نداء يخترق حد المعدل:** الكود الآن يحوّل تلقائياً للمزوّد الاحتياطي
  (`chat_completion` يلتقط UpstreamServiceError مرة واحدة). لو كلا المفتاحين
  مقيّداً، باعِد بين النداءات (~6 ثوانٍ) بدل الاعتماد على تراجع 429.
- شغّل **متسلسلاً**، وسجّل لكل سؤال: `qid, question, answer, top_chunks[أسماء
  الملفات], source, latency_ms`.

## 4) صيغة تقرير المقارنة المطلوبة

لكل سؤال: `qid | حكم قبل | حكم بعد | Δ (تحسّن/ثبات/تراجع) | زمن قبل | زمن بعد |
الملفات المسترجعة`. ثم ملخص:
- عدد: تحسّن / ثبات صحيح / ثبات خاطئ / **تراجع (الأهم — راقبه بدقة)**.
- متوسط الزمن قبل/بعد وP95 (هل التحسينات أبطأت الاستجابة؟ يجب لا).
- تصنيف الفشل المتبقي: استرجاع / توليد / فجوة بيانات / أمان.

**لا تعتمد نجاح حالة واحدة دليلاً، ولا تفعّل Reranker على Render قبل أن
تُظهر المقارنة تحسّناً يفوق كلفة الـ0.3-0.6 ثانية.**

## 5) مبادئ لا تُكسَر (من دليل RAG المشترك ومن قراراتنا)

- `app/config.py` هو المكان الوحيد الذي يقرأ `.env`.
- فلترة الصلاحيات (`allowed_collections`) قبل الترتيب دائماً — لا Metadata

- «البوت صديق الطالب»: يجيب الأسئلة الخارجية العامة بأدب — قرار منتج معتمد،
  ليس خطأً.
