**الغرض**: أن يفهم أي إنسان أو نموذج ذكاء اصطناعي (sol وأمثاله) المشروع فهماً
كاملاً: كيف تتواصل الملفات، ما رحلة السؤال، ما كل ثابت وعتبة ولماذا قيمته هكذا،
وكيف تعدّل بأمان. آخر تحديث: 17/7/2026 عند كوميت `5a116f7`.

> قاعدة ذهبية للمعدِّل: `app/config.py` هو **المكان الوحيد** الذي يقرأ `.env`.
> أي ثابت تبحث عنه يبدأ من هناك. وأي تعديل ⇒ شغّل
> `PYTHONIOENCODING=utf-8 python -m pytest tests/ -q` (يجب 316 ✅)،

## 1) الصورة الكبرى في ثلاث جمل

بوت RAG عربي: يخزّن معرفة الجامعة كملفات JSON في MongoDB، يقطّعها «مقاطع»
لكل منها متجه دلالي (Jina) وفهرس لفظي (BM25)، وعند كل سؤال يسترجع أفضل
المقاطع **بعد ترشيحها بصلاحية السائل** ويرسلها مع السؤال إلى LLM
(OpenRouter، متوافق OpenAI) ليصوغ الجواب تحت قواعد برومت صارمة.
الهوية عبر JWT بأربعة أدوار (زائر/طالب/موظف/أدمن)، ولوحة أدمن تدير الملفات
بنسخ ونشر ورجوع، والواجهة HTML/JS خفيفة يقدّمها نفس الخادم على `/app/`.

## 2) خريطة تواصل الملفات (من يستدعي من)

```
.env ← يُقرأ فقط في → app/config.py ← يستورده الجميع

الطبقة الدنيا (أدوات لا تعرف بعضها):
  app/log.py        تسجيل موحد
  app/errors.py     استثناءات النطاق (UpstreamServiceError…)
  app/text_norm.py  تطبيع عربي + tokenize (أساس كل مطابقة نصية)
  app/db.py         اتصال Mongo الوحيد (قاعدتان: iug_chatbot + uploaded_files)

طبقة المحركات:
  app/embeddings.py   عميل Jina + كاش متجهات الأسئلة (نص←متجه)
  app/lexical.py      BM25 صافي بايثون (يستعمل text_norm)
  app/retrieval.py    دمج الترتيبين dense+BM25 بمعادلة RRF
  app/chunking.py     وثيقة JSON ← نص مقطع واحد بترويسة [ملف: الاسم]
  app/index_store.py  حفظ/تحميل مصفوفات المتجهات (قرص أو Mongo، ببصمة نموذج+نصوص)
  app/llm.py          نداء LLM (retry مضاعف للطول، بث SSE، أخطاء باسم المزود)
  app/query_rewrite.py  كل ذكاء إعادة صياغة الاستعلام (نصي بحت، صفر LLM):
                        توسيع «قسمي»، سلسلة السياق، مصطلحات العامية،
                        نية القبول، الإحالات (اذكرهم/هذا…)
  app/sessions.py     ذاكرة المحادثة (Mongo أو ذاكرة): FIFO 5 أدوار بمتجهاتها
  app/cache.py        TTLCache عام (يستعمله chatbot للكاشين)
  app/ratelimit.py    نافذة معدل ثابتة thread-safe

طبقة الأمان والهوية:
  app/auth.py       حسابات students_auth (bcrypt) لكل الأدوار + bootstrap أدمن
  app/tokens.py     إصدار/تحقق JWT (HS256، sub+role+token_version)
  app/rbac.py       Principal + Role + can_read_student
  app/privacy.py    حارس «سؤال عن طالب آخر» + اختصار «حالتي الأكاديمية»
  app/context_builder.py  بناء السياق الخاص (ملف الطالب/الموظف) نصاً للبرومت
  app/file_catalog.py     سجل الملفات: تصنيف/أدوار/مالك/نسخ/نشر/رجوع/تبني
  app/admissions.py       كتالوج حقائق القبول الرقمية من الملفات (للمستخلص)
  app/audit.py            سجل تدقيق عمليات الأدمن

القلب:
  app/knowledge_base.py   فهرس القاعدة الرئيسية (شبه فارغة الآن — كل المحتوى ملفات)
  app/uploaded_files.py   UploadedFilesStore: تحميل/فهرسة/بحث كل الملفات + الحذف الكامل
  app/chatbot.py          IUGChatbot: يخيط كل ما سبق في رحلة السؤال (القسم 3)
  app/prompts.py          نصا البرومت — **نصهما هو سلوك البوت**، أي تعديل = تغيير سلوك

طبقة الويب:
  app/api/__init__.py   create_app: يجمع الراوترات + يقدّم frontend على /app (no-store)
  app/api/schemas.py    نماذج الطلب/الرد (حدود الأحجام هنا)
  app/api/deps.py       الحقن: get_bot، الهوية من التوكن، حدود المعدل، مفتاح الأدمن
  app/api/middleware.py CORS + GZip انتقائي (يستثني /stream) + ترويسات أمنية
  app/api/errors.py     صيغة الخطأ الموحدة {success:false, error:{...}}
  app/api/routers/      auth, chat, files, admin, sessions, health
  server.py             نقطة التشغيل (uvicorn)

الواجهة frontend/ (HTML/CSS/JS نقية RTL):
  index.html+app.js     دخول/تسجيل ← يخزن iug_auth في sessionStorage ← chat.html
  chat.html+chat.js     الشات: موثّق ← بث /student/stream؛ زائر ← /guest
                        مع guestHistory (آخر 5 أدوار في ذاكرة الصفحة فقط)
  admin.html+admin.js   لوحة الملفات (رفع/نشر/صلاحيات/حذف/نسخ)
  employee.html+…       بوابة الموظف
```

**قاعدة الاتجاه الواحد**: الطبقات تستورد ممن تحتها فقط. `chatbot.py` لا يعرف
FastAPI، والمحركات لا تعرف بعضها، والجميع لا يقرأ `.env` مباشرة.

## 3) رحلة السؤال خطوة بخطوة (المسار الأهم: chat_as_principal)

كل نقاط الشات تصب في `chatbot.chat_as_principal(question, principal,
allowed_collections, client_history)`:

1. **ترشيح الصلاحية قبل كل شيء**: الراوتر يحسب
   `file_catalog.allowed_collections(principal, الموجود)` — قائمة الملفات التي
   يحق للدور رؤيتها. كل بحث لاحق محصور فيها **بنيوياً** (متجهات ملف محظور لا
   تدخل الحساب أصلاً).
2. **حارس الخصوصية** (`privacy.py`): سؤال عن معدل/ترتيب طالب آخر ← رفض فوري
   بلا LLM. وسؤال الطالب عن «حالتي» ← إجابة مباشرة من ملفه بلا LLM.
3. **اختصارات فورية**: حقيقة موثوقة (عاصمة فلسطين/رابط الجامعة — مع استثناء
   أسئلة «الموقع الجغرافي/العنوان») أو حل قبول منظم واثق ← جواب فوري.
4. **الكاش**: يُقرأ فقط إذا كان الدور عاماً (لا سياق خاص) ولا سجل محادثة ولا
   إشارة عائدة في السؤال (`has_reference_tokens`). المفتاح: نوع المسار +
   السؤال + نطاق الملفات المسموح.
5. **بناء استعلام البحث** (نصي بحت في `query_rewrite.py`):
   - `base_question` = السؤال + مصطلحات العامية القانونية («اجل»←«تأجيل الدراسة»).
   - `search_question` = فوقه سلسلة السياق: الأسئلة القصيرة (≤3 توكنات) أو ذات
     الإحالات ترث نص الدور السابق **الطازج** (أحدث من 45 دقيقة).
6. **البحث المزدوج**: يُبحث بـ`search_question`، وإذا اختلف عن `base_question`
   يُبحث بالخام أيضاً و**تتقدم نتائجه** (موضوع السائل الحالي لا يُزاحَم)، مع
   سقف لإضافات السياق = `max(2, TOP_K//2)` — إلا سؤال الإحالة الخالص
   («اذكرهم») فبحثه الخام ضجيج ويُتخطى.
7. **بحث كل ملف**: لكل ملف مسموح: cosine على متجهاته + BM25، ثم دمج عالمي
   واحد عبر RRF وأخذ أفضل TOP_K فوق SIM_THRESHOLD.
8. **مسار القبول التجميعي**: إن حمل سؤال السائل نفسه نية قبول
   (`has_admission_intent` على `base_question`؛ وللمتابعة الخالصة على
   الموسّع): تُرسل ملفات القبول/المعدلات **كاملة** (حتى
   ADMISSION_TABLE_MAX_CHUNKS) + كتلة «مستخلص مفاتيح» نظيفة من
   `admissions.py` (كلية|برنامج|فروع|حد أدنى — بلا رسوم) كمرجع حصري للأرقام.
9. **الذاكرة**: للموثّق: من `sessions.py` الدور الأخير دائماً + الأدوار
   المشابهة (cosine ≥ MEMORY_MIN_SIM) من آخر 5. للزائر: الأدوار الخمسة
   القادمة من متصفحه (مقصوصة 500/1200 حرفاً) تُطوى كلها — ولا تُخزَّن أبداً.
10. **البرومت النهائي**: قالب `UPLOADED_FILE_SYSTEM_PROMPT` (هوية + 16 قاعدة
    سلوك) + المقاطع + مذكرة «الأحدث يفوز» عند تعدد المصادر + تعليمات شرطية
    (قبول/رسوم هندسة) + السياق الخاص للموثّق + الذاكرة، ثم LLM
    (بث تدريجي للموثّق، دفعة واحدة للزائر).
11. **بعد الجواب**: تنظيف جداول Markdown حتمياً (خادم `_strip_markdown_tables`
    + واجهة `tablesToLists`)، حفظ الدور بالذاكرة (لغير الزائر)، وتعبئة الكاش
    إن كان مؤهلاً.

## 4) جدول الثوابت والعتبات الكامل (القيم الفعلية وأثر تغييرها)

| الثابت | القيمة | أين | ماذا يفعل / أثر التغيير |
|---|---|---|---|
| `TOP_K` | 10 | config | عدد المقاطع للبرومت. رفعه = تغطية أوسع لكن ضجيجاً وتكلفة أكبر؛ لا ترفعه لحل سؤال تجميعي — لذلك مسار القبول |
| `SIM_THRESHOLD` | 0.25 | config | أرضية صلة المقطع؛ رفعها كثيراً = «غير وارد في بياناتي» زائفة |
| `RRF_K` | 60 | config | تخميد دمج الترتيبين (قياسي — لا يُلمس عادة) |
| `BM25_K1` / `BM25_B` | 1.5 / 0.75 | config | إشباع تكرار الكلمة / تطبيع الطول (قياسية) |
| `HYBRID_ENABLED` | true | config | إطفاؤه = دلالي فقط، تضيع الأرقام والأسماء الحرفية |
| `ADMISSION_TABLE_MAX_CHUNKS` | 150 | config | سقف إرسال جدول المفاتيح كاملاً؛ فوقه يعود لبحث موسع |
| `MAX_HISTORY` | 5 | config | أدوار الذاكرة FIFO |
| `MEMORY_MIN_SIM` | 0.45 | config | عتبة «الدور السابق ذو صلة»؛ خفضها = حقن ثرثرة، رفعها = فقدان سياق |
| `MEMORY_FRESH_MINUTES` | 45 | config | نضارة السلسلة: متابعة قصيرة لا ترث دوراً أقدم من هذا |
| `HISTORY_TURNS_IN_PROMPT` | 6 | config | حد أدوار السجل المطوية نصاً في البرومت |
| `_SHORT_QUESTION_TOKENS` | 3 | query_rewrite | «قصير» = ≤3 توكنات: عتبة السلسلة والإحالة الخالصة |
| `_ANAPHORA_TOKENS` | قائمة | query_rewrite | كلمات الإحالة (اذكرهم/هذا/كمان…): تشعل السلسلة وتمنع الكاش |
| قص سجل الزائر | 500/1200 حرفاً | routers/chat.py | سؤال/جواب الدور المطوي — علاج تدهور المحادثات الطويلة |
| حدود GuestTurn | 2000/6000 حرفاً و≤5 أدوار | schemas | ما يقبله الخادم من المتصفح أصلاً |
| حد السؤال | 2000 حرفاً | schemas | فوقه 422 |
| `LLM_MAX_TOKENS` | 450 | config | سقف الجواب؛ عند بتر بمحتوى يعاد النداء بضعفه (llm.py) |
| `LLM_TEMPERATURE` | 0.05 | config | شبه حتمي — مطلوب لبوت حقائق |
| `LLM_REASONING_EFFORT` | low | config | لموديلات gpt-oss: لا يأكل التفكير ميزانية الجواب |
| مهلة/محاولات LLM | 60 ثانية / 4 | llm.py | إعادة محاولة بتراجع عند أخطاء المزود |
| `CACHE_ANSWER_TTL/MAXSIZE` | 3600 ث / 512 | config | كاش الإجابات العامة فقط (شروط القسم 3-4) |
| `CACHE_EMBED_TTL/MAXSIZE` | 86400 ث / 2048 | config | كاش متجهات الأسئلة (نص←متجه، حتمي) |
| `RATE_LIMIT_CHAT_PER_MIN` | 30 | config | لكل هوية (وللزائر لكل IP — يتطلب `--proxy-headers` خلف وكيل، موجود في render.yaml) |
| `RATE_LIMIT_LOGIN_PER_MIN` | 10 | config | login/register فقط (لا /me) |
| `JWT_EXPIRE_HOURS` | 12 | config | عمر التوكن؛ الإبطال الفوري عبر token_version في الحساب |
| `EMBED_BATCH_SIZE` | 64 | config | دفعات نداء Jina عند الفهرسة |
| `INDEX_BACKEND` | disk محلياً / mongo على Render | config | أين تُحفظ المتجهات؛ أسماء ملفات القرص SHA1 في `.index_cache/` |
| `_ALWAYS_EXCLUDE_FROM_RAG` | 8 مجموعات | config | حسابات/جلسات/فهارس **لا تُفهرس أبداً** — خط أحمر أمني |
| تصنيفات الملفات | 5 | file_catalog | university_public (الكل) ← student_records (طالب+موظف+أدمن) ← employee_internal/private ← admin_only؛ الكتابة والقراءة مقصوصتان على سقف التصنيف |

## 5) قرارات منتج معتمدة (لا «تصلحها»)

- **البوت صديق الطالب**: السؤال خارج نطاق الجامعة (طبخ، معلومة عامة…) يُجاب
  من المعرفة العامة بأدب — قرار معتمد نهائياً (17/7/2026). الممنوع فقط:
  اختراع معلومات جامعية، ووصف جامعات أخرى، وعاصمة فلسطين هي القدس.
- **الزائر بلا جلسات على الخادم** (خصوصية): ذاكرته في متصفحه فقط.
- **«الأحدث يفوز»** عند تعارض ملفين: مذكرة تواريخ للنموذج، وليست حذفاً للقديم.
- الأخطاء بعد أول بايت في البث تظهر نصاً داخل البث (مقايضة موثقة).

## 7) مناطق الخطر (اقرأ قبل أن تلمس)

1. **`_ALWAYS_EXCLUDE_FROM_RAG`**: إزالة أي عنصر منها = تسريب هاشات كلمات
   مرور وجلسات إلى فهرس البحث.
2. **ترشيح الصلاحية قبل البحث** (`allowed_collections` في كل مسار): أي مسار
   جديد يجب أن يمرر القائمة — نسيانها يفتح ملفات خاصة للجميع.
3. **شروط الكاش الأربعة** (عام + بلا سجل + بلا سياق خاص + بلا إحالة): إضافة
   شرط تخزين جديد يجب ألا يكسر «رد خاص لا يُكاش أبداً».
4. **اسم فهرس المتجهات** موحد عبر `_index_name()` = `uploaded::الاسم` — البناء
   والحذف يستعملانها معاً (اسم مجرد سبب يتيمة تاريخياً).
5. **مسار جانبي جديد** (استيراد/تبني/ترحيل) يجب أن ينتج نفس آثار المسار
   الطبيعي كاملة (كتالوج + لقطة نسخة + فهرس) — درس الثقب 36.
6. **حارس/قيد جديد** يُرفق بتنظيف رجعي للبيانات المخالفة الموجودة — درس 37.
7. **الواجهة**: أي HTML يُبنى من نص خارجي يمر بـ`escapeHtml` أولاً
   (`renderMarkdown` في chat.js) — لا `innerHTML` مباشرة أبداً.
8. **عملية واحدة**: الكاش وحدود المعدل في الذاكرة — تشغيل عدة workers يكسرهما
   (المطلوب حينها Redis؛ موثق كقرار مؤجل).
