# تنفيذ الخطة المدمجة — Adaptive RAG v2

## النتيجة التنفيذية

تم دمج نقاط القوة من خطتين:

1. **خطة الإصلاح منخفضة المخاطر**: توحيد البرومت، إزالة التعارضات، تحسين فهم الحوار، الأدلة المنظمة، التحقق، والـReranker الانتقائي.
2. **خطة المعمارية المستقبلية**: QueryPlan، Structured Router، Contextual/Hierarchical Chunking، مرشحون بدرجات كاملة، Evidence Coverage، وقاطع دائرة للـReranker.

القيد المعتمد في هذه النسخة:

> كل سؤال مستخدم يصل إلى الـLLM لصياغة الجواب النهائي. لا يوجد Trusted Fact أو Structured Resolver أو Privacy Shortcut أو Answer Cache يعيد جوابًا نهائيًا دون استدعاء الـLLM.

الاسترجاع والخدمات الحتمية ما زالت تُستخدم **لتجهيز الأدلة فقط**، لأن إلغاؤها كليًا يعني إرسال كل قاعدة المعرفة إلى النموذج أو تركه يجيب بلا مصادر. لكنها لا تجيب المستخدم مباشرة.

---

## مسار السؤال الجديد

```text
سؤال المستخدم
  ↓
صلاحيات وخصوصية قبل الوصول إلى البيانات
  ↓
ConversationFrame + QueryPlan (حتمي، بلا LLM إضافي)
  ↓
Domain Router
  ↓
حقائق منظمة + Hybrid Retrieval عند الحاجة
  ↓
Dense + BM25 + RRF
  ↓
Selective Reranker عند الحاجة فقط
  ↓
Contextual/Hierarchical Evidence
  ↓
EvidenceContract + Retry واحد للحقل الناقص
  ↓
برومت موحد
  ↓
LLM — الجواب النهائي لكل سؤال
  ↓
Answer Check وفallback آمن بعد استدعاء النموذج إذا لزم
```

## 1. توحيد البرومت

تم استبدال نسختي البرومت المتكررتين بمنشئ واحد:

- `PromptRoute`
- `PromptContext`
- `build_system_prompt(...)`

ترتيب الأولوية أصبح صريحًا:

1. الأمان والخصوصية.
2. طلب المستخدم وتصحيحاته الأخيرة.
3. حالة الحوار.
4. الحقائق المنظمة وعقد الأدلة.
5. المقاطع المسترجعة.
6. أسلوب العرض.

تم حذف حد **300 حرف** واستبداله بقاعدة:

> الإجابة موجزة افتراضيًا، ويجوز الإطالة بقدر إكمال قائمة أو مقارنة أو سؤال متعدد الأجزاء.

كما تم توحيد قواعد:

- عدم تعويض المعلومة المطلوبة بمعلومة مجاورة.
- فصل أنواع الرسوم.
- فصل القبول عن المنح والمعدل الجامعي.
- فصل البرنامج عن المسار.
- فصل البوابة عن النموذج ونظام التعليم الإلكتروني.
- عدم وصف الحقائق الثابتة بأنها موسمية.

## 2. ConversationFrame وQueryPlan

أضيفت طبقة حتمية لا تستخدم نموذجًا إضافيًا، وتقرأ كلام المستخدم فقط.

تتعقب:

- النية.
- المجال.
- الموضوع الحالي.
- المرجع.
- المرحلة الأكاديمية.
- فرع الثانوية.
- المعدل ونوعه.
- نطاق التحويل.
- الحقول المطلوبة.
- المواضيع المستبعدة صراحة.
- هل السؤال تابع أو مبهم أو مركب أو يطلب قائمة.

الأحدث من تصحيحات المستخدم يلغي الأقدم.

تم إصلاح حالة مهمة:

> عبارة «مواد علمية» لا تعني أن فرع الثانوية علمي.

---

## 3. الاستبعادات والاستعلام الإيجابي

أضيف `positive_query(...)` لفصل المطلوب الإيجابي عن العبارات المستبعدة.

مثال:

```text
خلينا من الهندسة، شو منح المتفوقين؟
```

يصبح استعلام البحث:

```text
شو منح المتفوقين؟
```

مع الاحتفاظ بـ`الهندسة` داخل قائمة الاستبعادات حتى لا تعود أثناء الاسترجاع أو Retry استكمال الأدلة.

السؤال الأصلي نفسه لا يُغيّر عند إرساله إلى الـLLM أو تخزينه في السجل.

---

## 4. Structured Router

أضيف `DomainRoute` لتحديد طريقة تجهيز الأدلة:

- `privacy`
- `structured_plus_rag`
- `compound`
- `general_rag`

الـRouter لا ينتج جوابًا، بل يقرر:

- هل نحتاج حقائق منظمة؟
- هل نحتاج استرجاعًا واسعًا؟
- هل نحتاج Reranker؟
- ما سبب اختيار المسار؟

القبول المنظم والحقائق المرجعية أصبحت أدلة داخل البرومت، وليست إجابات نهائية.

---

## 5. EvidenceContract

أضيف عقد أدلة حتمي يحدد:

- الحقول المطلوبة.
- الحقول المسندة.
- الحقول الناقصة.
- التعارضات.
- هل الأدلة كافية؟
- هل السؤال حساس للحقائق والأرقام؟

عند وجود جزء ناقص، يسمح النظام بـRetry واحد محدود للحقل الناقص فقط، ثم يطلب من الـLLM:

- الإجابة بالأجزاء المسندة.
- التصريح بنقص البقية.
- عدم التخمين.

كما يكشف تعارضات مثل:

- دليل ماجستير لسؤال بكالوريوس.
- معدل جامعي في سؤال عن معدل الثانوية.
- تحويل خارجي في سؤال عن تحويل داخلي.

---

## 6. Contextual وHierarchical Chunking

أضيف نموذج `ChunkRecord` مع:

- `chunk_id`
- `parent_id`
- `source`
- `doc_index`
- `kind`
- `parent_text`
- `metadata`

الوثائق التي تحتوي قوائم من عناصر فرعية تُقسم إلى:

```text
Parent / Overview
├── Child 1
├── Child 2
└── Child n
```

كل Child يحتفظ بسياق الـParent، لذلك يمكن البحث في معلومة دقيقة دون فقدان:

- اسم الكلية.
- المرحلة.
- نوع المعلومة.
- الوثيقة الأصلية.

معرفات الـChunks ثابتة لنفس المحتوى والبنية.

---

## 7. RetrievalCandidate

لم يعد المسار المتقدم يتعامل مع النص وحده، بل مع مرشح يحتوي:

- Dense Score.
- BM25 Score.
- RRF Score.
- Fused Rank.
- Source.
- Chunk ID.
- Parent ID.
- Metadata.

هذا يسمح بالتشخيص والتقييم دون إعادة الاسترجاع.

في واجهة V2 لا يتم إجبار أقرب Chunk ضعيف على الدخول إذا لم ينجح لا دلاليًا ولا لفظيًا. عند غياب دليل قوي، يخبر EvidenceContract النموذج أن الأدلة ناقصة.

واجهة الاسترجاع القديمة بقيت للتوافق مع الأجزاء القديمة والاختبارات.

---

## 8. Selective Reranker

الـReranker يعمل فقط عندما يقرر QueryPlan وDomainRoute أن السؤال يحتاجه، مثل:

- سؤال تابع أو مبهم.
- سؤال مركب.
- سؤال يحتاج دليلًا دقيقًا.
- نتائج متنافسة في مجال غير منظم.

لا يُستخدم كبديل للحقائق المنظمة والجداول الحسابية.

أضيف:

- Timeout افتراضي: `2.5s`.
- Fail-open إلى ترتيب RRF.
- Circuit Breaker بعد فشلين.
- Cooldown قبل إعادة المحاولة.
- Metadata صريحة عن applied/fallback/timeout/circuit-open.

---

## 9. كل سؤال يصل إلى الـLLM

تم تحويل المسارات التالية من إجابات مباشرة إلى أدلة للـLLM:

- الحقائق المرجعية الموثوقة.
- نتيجة القبول المنظمة.
- رفض طلب بيانات طالب آخر.
- بيانات الطالب المصرح بها.
- حالة الملف غير الموجود.
- أسئلة المصدر والتاريخ.
- المسار العادي والمسار المتدفق Streaming.

تم فصل كاش الإجابة النهائية عن كاش الـEmbeddings:

```env
LLM_ALWAYS_ANSWER=true
ANSWER_CACHE_ENABLED=false
```

- **كاش Embeddings** يبقى مفعلًا لتحسين السرعة.
- **كاش الجواب النهائي** معطل افتراضيًا حتى لا يتجاوز أي سؤال استدعاء الـLLM.

قد يعمل Answer Check بعد جواب النموذج ويستبدل جوابًا خطيرًا أو مخمّنًا بـFallback آمن، لكن ذلك يحدث **بعد استدعاء الـLLM**، وليس بدلًا عنه.

## 13. ما لم يتم تشغيله

لم يتم تشغيل اختبار الـ408 سؤالًا عبر مزود LLM الحقيقي، لأن ذلك يستهلك API فعليًا وقد يغيّر بيانات أو تكاليف التشغيل. الاختبارات المنفذة حتمية وتستخدم Mock للـLLM.

التشغيل الحي المقترح لاحقًا يكون بجولتين:

1. جودة مع Answer Cache معطل.
2. أداء وكاش بشكل منفصل.

مع التحقق من:

```text
440 total / 32 excluded / 408 evaluated
```

## 14. الخلاصة

هذه النسخة تجمع بين إصلاحات المشروع الحالية والمعمارية الأقوى للمستقبل:

```text
Prompt Cleanup
+ ConversationFrame
+ QueryPlan
+ Structured Router
+ EvidenceContract
+ Contextual/Hierarchical Chunking
+ Dense/BM25/RRF Candidates
+ Selective Reranker
+ LLM Final Answer for Every Question
```

لم يعد أي Resolver أو Shortcut يقرر جواب المستخدم النهائي. كل مكون قبل الـLLM يجهز الأدلة والقيود فقط، بينما يبقى الـLLM هو نقطة صياغة الجواب النهائية لكل سؤال.
