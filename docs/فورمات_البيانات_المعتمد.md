# الفورمات المعتمد لملفات معرفة الشات بوت (JSON)

هذه المواصفة مبنية على **كيف يعمل محرك البوت فعلياً** — لا على تفضيلات شكلية.

## كيف يهضم المحرك ملفك (مهم لفهم كل قاعدة هنا)

1. **كل عنصر في المصفوفة = مقطع بحث واحد**: يُسطَّح العنصر كاملاً إلى نص
   («المفتاح: القيمة» سطراً سطراً) ويُبنى له متجه دلالي + فهرس لفظي BM25.
2. **لا يوجد نظام «كلمة مفتاحية ← جواب آلي»**: حقل `keywords` لا يفعّل أي
   مطابقة مباشرة — يُسطَّح نصاً فيُحسّن فرص الاسترجاع فقط. **التوليد دائماً
   عبر النموذج** الذي يقرأ المقاطع ويصيغ ويطبّق قواعد البرومت (يفكّر).
3. حقول مثل `embedding_text` و`intents` و`must_be_applied_by_code` **ليست
   حقولاً خاصة عند المحرك** — تُعامل نصاً عادياً داخل المقطع.
4. سلوك البوت (فهم السياق، حل «اذكرهم»، عدم القفز للمودل، التوجيه للجهات)
   **مبنيّ في الكود والبرومت** — لا تكتبه كوثائق بيانات.

## المواصفة الذهبية

**الجذر: مصفوفة JSON** `[ {...}, {...} ]` — أبداً لا كائن واحد يلفّ كل شيء
(كائن واحد = مقطع عملاق واحد يفسد الاسترجاع).

**كل عنصر = وحدة معرفية ذرّية واحدة** (سؤال واحد / حقيقة واحدة / إجراء واحد)،
بحجم **300–1500 حرف** بعد التسطيح.

### قالب العنصر

```json
{
  "الموضوع": "رسوم طلب الالتحاق للبكالوريوس",
  "التصنيف": "الرسوم",
  "صيغ_السؤال": [
    "كم رسوم طلب الالتحاق للبكالوريوس؟",
    "بدي اقدم طلب التحاق، كم بيكلفني؟",
    "كم بدفع عشان اسجل طلب جديد؟"
  ],
  "الإجابة": "رسوم طلب الالتحاق لبرنامج البكالوريوس هي 20 ديناراً أردنياً، تُدفع مرة واحدة عند التقديم. للتفاصيل: عمادة القبول والتسجيل regist@iugaza.edu.ps أو https://admission.iugaza.edu.ps/",
  "حقائق": [
    "الرسم يُدفع مرة واحدة فقط",
    "يختلف عن الرسوم الدراسية (سعر الساعة)"
  ],
  "الجهة_المختصة": "عمادة القبول والتسجيل",
  "المصدر": "https://admission.iugaza.edu.ps/",
  "آخر_تحقق": "2026-07-15"
}
```

### قواعد الحقول

| الحقل | القاعدة |
|-------|---------|
| `الموضوع` | عنوان محدد يميّز العنصر (يظهر في المقطع ويقوّي ترتيبه) |
| `صيغ_السؤال` | **3–6 أسئلة كاملة طبيعية** (فصحى + عامية غزّية) — أقوى أثر استرجاعي في الملف كله؛ هذه بديل «keywords» الصحيح |
| `الإجابة` | جواب نهائي مكتمل بصياغة تصل الطالب كما هي تقريباً: يتضمن الرقم بعملته (**دينار أردني**)، الرابط الكامل، والجهة — لا إحالات لمعرّفات |
| `حقائق` | نقاط قصيرة اختيارية تعزز الدقة |
| `الجهة_المختصة` | «عمادة القبول والتسجيل» أو «عمادة شؤون الطلبة» — يرسّخ التوجيه الصحيح |
| `المصدر` + `آخر_تحقق` | للمراجعة البشرية ولترجيح الأحدث |

### الممنوعات (كل واحدة سببت مشكلة حقيقية)

1. ❌ **كلمات مفتاحية مفردة عامة** («كم بكلف»، «الدفع»، «شو أعمل», «المعبر مغلق»
   في عنصر غير مخصص لها) — تسحب المقطع لأسئلة ليست له وتلوّث استرجاع غيره.
   البديل: صيغ سؤال كاملة خاصة بالعنصر نفسه.
2. ❌ **روابط بمعرّفات مرجعية** (`"link_id": "student_portal"` مع قسم links
   منفصل) — المحرك لا يفكّ المرجع؛ ضع الـURL الكامل داخل العنصر نفسه.
3. ❌ **عنصر يجمع مواضيع متعددة** أو كائن جذري واحد يلفّ الملف.
4. ❌ **سياسات تشغيل داخل ملفات البيانات** (`runtime_policy` /
   `must_be_applied_by_code` / خوارزميات فهم السياق): المحرك لا ينفّذها،
   وقد تُسترجع كنص فتزاحم المعرفة الحقيقية. سلوك البوت مكانه البرومت
   والكود — **أرسلوا القاعدة السلوكية لمحمود/كلود لتُبنى في النظام**.
   (فهم «اذكرهم»، حل الضمائر، عدم القفز للمودل، التحقق من نوع الكيان —
   كلها مبنية فعلاً في المحرك اليوم.)
5. ❌ جداول Markdown داخل `الإجابة` (تُحوَّل قسرياً لقوائم — اكتبها قوائم من البداية).
6. ❌ تكرار المعلومة نفسها في عناصر كثيرة — نسخ متطابقة تحتل مقاعد
   الاسترجاع العشرة وتزاحم معلومات أخرى. عنصر «قائمة مجمّعة» واحد + عنصر
   لكل مفردة يكفي (كما في دليل الكليات — جيد).

### برومت توليد جاهز (الصقه في أي نموذج ذكاء اصطناعي مع المحتوى الخام)

> حوّل المحتوى التالي إلى مصفوفة JSON لقاعدة معرفة شات بوت الجامعة الإسلامية (غزة).
> القواعد: (1) الجذر مصفوفة `[]`؛ (2) كل عنصر وحدة معرفية واحدة بالحقول:
> الموضوع، التصنيف، صيغ_السؤال (3-6 أسئلة كاملة فصحى وعامية فلسطينية)،
> الإجابة (نهائية مكتملة، المبالغ بالدينار الأردني، الروابط كاملة داخلها،
> مع ذكر الجهة المختصة: القبول والتسجيل لأمور التسجيل/الجداول/الوثائق،
> وشؤون الطلبة للمنح والمساعدات والمشاكل)، حقائق (نقاط قصيرة)،
> الجهة_المختصة، المصدر، آخر_تحقق؛ (3) حجم العنصر 300-1500 حرفاً؛
> (4) ممنوع: كلمات مفتاحية مفردة عامة، معرّفات روابط، جداول Markdown،
> تعليمات سلوك للبوت، دمج مواضيع متعددة في عنصر؛ (5) لا تخترع أي رقم أو
> رابط غير موجود في المحتوى الخام — إن غاب فاكتب في الإجابة أنه يُتحقق منه
> من الجهة المختصة. أخرج JSON فقط.

## تقييم الملفين المرفوعين (2026-07-15)

**ما رُفع فعلاً مقسّم جيداً** (15 و16 مقطعاً بأحجام صحية) وأجاب ممتازاً في
بنك الاختبار. التقييم التفصيلي:

| الجانب | التقييم |
|--------|---------|
| `answer` جاهزة داخل كل عنصر | ✅ ممتاز — هذا أهم حقل، أبقوه دائماً |
| عنصر لكل كلية + عنصر القائمة الكاملة (`complete_list`) | ✅ ممتاز — حلّ مشكلة «القائمة الجزئية» |
| أمثلة محادثة (`response_examples`) | ✅ جيدة بجرعة صغيرة |
| `keywords` | ⚠️ تعمل كنص استرجاعي فقط (لا «إجابة بلا تفكير») — لكن العامة منها («كم بكلف»، «الدفع») احذفوها واستبدلوها بصيغ سؤال كاملة |
| `embedding_text` / `intents` | ⚠️ لا يعاملها المحرك معاملة خاصة — نص إضافي غير ضار لكنه وزن زائد؛ يمكن حذفها |
| عناصر السياسة (`runtime_policy`، `assistant_policy`) | ❌ المحرك لا ينفّذها كتعليمات — والمطلوب فيها **مبنيّ أصلاً في الكود** (حل الضمائر، النضارة، منع القفز للمودل). احذفوها من ملفات البيانات، وأي قاعدة سلوك جديدة ترسل لتُبنى |
| قسم `links` بمعرّفات (نسخة سطح المكتب) | ❌ لا يُفكّ — النسخة المرفوعة عالجت هذا بوضع الروابط داخل الإجابات ✅ |
| كائن جذري واحد (نسخة سطح المكتب للملف الأول) | ❌ كان سيصبح مقطعاً واحداً عملاقاً — النسخة المرفوعة مصفوفة ✅ |
